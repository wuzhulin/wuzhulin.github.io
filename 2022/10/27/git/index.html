git版本控制工具

1 Git的概念

1.1 git与svn的区别

Git 不仅仅是个版本控制系统，它也是个内容管理系统(CMS)，工作管理系统等。

如果你是一个具有使用 SVN 背景的人，你需要做一定的思想转换，来适应 Git 提供的一些概念和特征。

Git 与 SVN 区别点：

  • 1、Git 是分布式的，SVN 不是：这是 Git 和其它非分布式的版本控制系统，例如 SVN，CVS 等，最核心的区别。
  • 2、Git 把内容按元数据方式存储，而 SVN 是按文件：所有的资源控制系统都是把文件的元信息隐藏在一个类似 .svn、.cvs 等的文件夹里。
  • 3、Git 分支和 SVN 的分支不同：分支在 SVN 中一点都不特别，其实它就是版本库中的另外一个目录。
  • 4、Git 没有一个全局的版本号，而 SVN 有：目前为止这是跟 SVN 相比 Git 缺少的最大的一个特征。
  • 5、Git 的内容完整性要优于 SVN：Git 的内容存储使用的是 SHA-1 哈希算法。这能确保代码内容的完整性，确保在遇到磁盘故障和网络问题时降低对版本库的破坏

1.2 集中化的管理系统

将所有的代码集中管理中服务器中，客户端只需要保存最新的代码版本，缺点就是服务器宕机就不能使用，著名的有svn，像svn的每个版本存储的都是上一个版本的差异

1.3 分布式的版本控制系统

每一个客户端都是一个服务器，都保存着完整的版本记录，这时一台服务器出现故障并不会影响到其他服务器。而且由于其优秀的解压算法，即便保存了所有版本也不占空间。git每一个版本都存储了前面版本的索引

2 Git的结构

​ 工作区（WORKING DIRECTORY）: 直接编辑文件的地方，肉眼可见直接操作；

　　暂存区（STAGIN AREA）：数据（快照）暂时存放的地方；

　　版本库（GIT DIRECTORT(RESPOSITORY)）：存放已经提交的数据，push 的时候，就是把这个区的数据 push 到远程git仓库了。

　　git add就是将工作区的修改缓存在暂存区，git commit就是将暂存区的数据快照提交到本地库

　　这就是为什么 git commit 之前要先执行 git add 的原因,如果不先执行add，那么直接执行commit时不会把当前的修改内容提交到代码库中的。

5 Git的常见命令

1 添加和提交

添加：将工作区的文件添加的暂存区

命令：git add【文件】

提交：将暂存区的文件提交到本地库

命令：git commit -m 【说明】【文件】

2 查看库中文件状态信息

状态：可以查看文件处于什么区中，更加明确是否已经提交

命令：git status

修改Demo1中的信息，再查看状态

修改后的文件需要重新添加和提交

3 查看提交日志

日志：可以查看提交的日志以及对应的索引，方便回退

命令：git log

当历史记录过多时，查看日志会有分屏的效果，分屏操作下：

  • 下一页：空格
  • 上一页：b
  • 退出：q

日志的其他展示方式：

1 git log –pretty=oneline

所有记录展示在一行

2 git log –oneline

3 git reflog

4 前进或后退历史版本

此功能可以让我们退回和前进到历史版本，当前的版本信息如下：

执行回退命令到创建文件的版本 ： git reset –hard 【索引】

此时我们在查看Text文件

此时该文件中添加的内容全部消失，回到了创建文件之初。

其他的回退操作

  1. –hard 是指将工作区、暂存区和本地库全部回退到指定的版本
  2. –mixed 可以不回退工作区
  3. –soft 只回退本地库

5 删除文件

删除文件大致分为以下三步：

  1. 删除工作区的文件

  2. 将删除的文件提交到暂存区

  3. 再提交到本地库

此时就完成了文件的删除

这样删除有一个好处就是可以回退恢复文件

只需要回退到删除文件之前就可以恢复文件了

6 查看差异

当工作区中的内容与暂存区中的内容不一致时，我们可以使用diff命令查看差异

此时就造成了工作区与暂存区的内容不一致

命令：git diff 【文件名】，当然也可以不指定文件名，会查找所有工作区与暂存区中所有的差异

也可以查看暂存区与本地库的区别：

命令：git diff HEAD 【文件名】

通过指定索引还可以查看工作区与不同版本的本地库之间的差异

命令: git diff 【索引】 【文件名】

6 分支

1 什么是分支

在版本控制过程中，使用多条线来同时推进多个任我。这里的多条线就是多个分支。

分支图：

2 分支在实际中有什么用呢？

假设你准备开发一个新功能，但是需要两周才能完成，第一周你写了50%的代码，如果立刻提交，由于代码还没写完，不完整的代码库会导致别人不能干活了。如果等代码全部写完再一次提交，又存在丢失每天进度的巨大风险。

现在有了分支，就不用怕了。你创建了一个属于你自己的分支，别人看不到，还继续在原来的分支上正常工作，而你在自己的分支上干活，想提交就提交，直到开发完毕后，再一次性合并到原来的分支上，这样，既安全，又不影响别人工作。

3 分支操作

3.1 查看分支

命令： git branch -v

3.2 创建分支

命令： git branch 【分支名】

3.3 切换分支

命令： git checkout 【分支名】

3.4 合并分支以及解决分支冲突

1.在分支文件中提交修改后的Text2

2.在master中提交修改后的Text2

3.在master中合并分支branch1，这时就会造成分支冲突，因为两个分支都修改了Text2文件

命令：git merge branch1

解决办法：人为修改冲突文件中的内容

4.然后再把修改后的文件提交

5.此时就完成了分支的合并功能

7 本地库与远程库的操作

1 创建远程库

  1. 登录gitHub

  2. 创建一个远程库

  3. 设置远程库信息

  4. 创建成功

2 给远程库地址取别名

1 获取远程库地址

2 在本地查看查看别名列表

3 设置别名

命令：git remote add 【别名】【远程库地址】 可以看到我们的别名已经取好了

3 将本地库推送到远程库

命令：git push 【别名|远程库地址】【分支名】

通过验证之后即可连接成功，远程库中便有了本地库的文件

4 将远程库克隆到本地库

先找到项目的链接，找到对应的目录执行clone操作

克隆操作会完成的操作：

  • 初始化本地库
  • 将远程库内容完整的克隆到本地
  • 替我们创建远程库的别名

5 将远程库拉取到本地库

命令 git pull 【远程库地址】

pull操作其实是fetch与merge的集合
